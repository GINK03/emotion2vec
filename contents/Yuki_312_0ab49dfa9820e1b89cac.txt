All current Low Energy application profiles are based on GATT.\n\nThe Bluetooth SIG defines many profiles for Low Energy devices. A profile is a specification for how a device works in a particular application. Note that a device can implement more than one profile. For example, a device could contain a heart rate monitor and a battery level detector.\n\n\nAttribute Protocol (ATT) \u2014GATT is built on top of the Attribute Protocol (ATT). This is also referred to as GATT/ATT. ATT is optimized to run on BLE devices. To this end, it uses as few bytes as possible. Each attribute is uniquely identified by a Universally Unique Identifier (UUID), which is a standardized 128-bit format for a string ID used to uniquely identify information. The attributes transported by ATT are formatted as characteristics and services.\nCharacteristic \u2014A characteristic contains a single value and 0-n descriptors that describe the characteristic's value. A characteristic can be thought of as a type, analogous to a class. \nDescriptor \u2014Descriptors are defined attributes that describe a characteristic value. For example, a descriptor might specify a human-readable description, an acceptable range for a characteristic's value, or a unit of measure that is specific to a characteristic's value.\nService \u2014A service is a collection of characteristics. For example, you could have a service called \"Heart Rate Monitor\" that includes characteristics such as \"heart rate measurement.\" You can find a list of existing GATT-based profiles and services on bluetooth.org.\n\n\n\u3053\u308c\u3089\u306fBLE\u306e\u30ad\u30fc\u30ef\u30fc\u30c9\u3068\u30b3\u30f3\u30bb\u30d7\u30c8\u3067\u3042\u308b. \n\n\nGeneric Attribute Profile (GATT) GATT\u30d7\u30ed\u30d5\u30a1\u30a4\u30eb\u306fBLE\u901a\u4fe1\u306b\u3088\u308b\u30c7\u30fc\u30bf\u306e\u9001\u4fe1\u3084\u65e2\u77e5\u306e\"attributes\"\u306e\u65ad\u7247\u3092\u53d7\u4fe1\u3059\u308b\u305f\u3081\u306e\u57fa\u672c\u4ed5\u69d8\u3067\u3042\u308b. \u5168\u3066\u306eBLE\u30a2\u30d7\u30ea\u30b1\u30fc\u30b7\u30e7\u30f3\u306e\u30d7\u30ed\u30d5\u30a1\u30a4\u30eb\u306fGATT\u30d7\u30ed\u30d5\u30a1\u30a4\u30eb\u304c\u30d9\u30fc\u30b9\u306b\u306a\u308b.\n\nBluetooth SIG\u3067\u306f\u3044\u3064\u304f\u304b\u306eBLE\u30d7\u30ed\u30d5\u30a1\u30a4\u30eb\u304c\u5b9a\u7fa9\u3055\u308c\u3066\u3044\u308b. \u30d7\u30ed\u30d5\u30a1\u30a4\u30eb\u306f\u30c7\u30d0\u30a4\u30b9\u304c\u7279\u5b9a\u306e\u30a2\u30d7\u30ea\u30b1\u30fc\u30b7\u30e7\u30f3\u3067\u3069\u306e\u3088\u3046\u306b\u52d5\u4f5c\u3059\u308b\u304b\u306e\u4ed5\u69d8\u3067\u3042\u308b. \u30c7\u30d0\u30a4\u30b9\u306f\u8907\u6570\u306e\u30d7\u30ed\u30d5\u30a1\u30a4\u30eb\u3092\u5b9f\u88c5\u3067\u304d\u308b\u70b9\u306b\u306f\u6ce8\u610f\u3059\u308b\u3053\u3068. \u4f8b\u3048\u3070\u3042\u308b\u30c7\u30d0\u30a4\u30b9\u306f\u5fc3\u62cd\u6570\u30e2\u30cb\u30bf\u30fc\u3068\u96fb\u529b\u6d88\u8cbb\u30e2\u30cb\u30bf\u30fc\u3092\u542b\u3080\u3053\u3068\u304c\u3067\u304d\u308b. \n\n\nAttribute Protocal (ATT) - GATT\u306fAttribute Prorotocol(ATT)\u306e\u4e0a\u4f4d\u306b\u7d44\u307f\u8fbc\u307e\u308c\u308b. \u3053\u308c\u306fGATT/ATT\u3068\u547c\u3070\u308c\u308b. This applies to the BLE connection itself. The device in the central role scans, looking for advertisement, and the device in the peripheral role makes the advertisement.\nGATT server vs. GATT client. This determines how two devices talk to each other once they've established the connection.\n\nTo understand the distinction, imagine that you have an Android phone and an activity tracker that is a BLE device. The phone supports the central role; the activity tracker supports the peripheral role (to establish a BLE connection you need one of each\u2014two things that only support peripheral couldn't talk to each other, nor could two things that only support central).\nOnce the phone and the activity tracker have established a connection, they start transferring GATT metadata to one another. Depending on the kind of data they transfer, one or the other might act as the server. For example, if the activity tracker wants to report sensor data to the phone, it might make sense for the activity tracker to act as the server. If the activity tracker wants to receive updates from the phone, then it might make sense for the phone to act as the server.\nIn the example used in this document, the Android app (running on an Android device) is the GATT client. The app gets data from the GATT server, which is a BLE heart rate monitor that supports the Heart Rate Profile. But you could alternatively design your Android app to play the GATT server role. All current Low Energy application profiles are based on GATT.\n>     - The Bluetooth SIG defines many [profiles](https://www.bluetooth.org/en-us/specification/adopted-specifications) for Low Energy devices. A profile is a specification for how a device works in a particular application. Note that a device can implement more than one profile. For example, a device could contain a heart rate monitor and a battery level detector.\n\n> - **Attribute Protocol (ATT)** \u2014GATT is built on top of the Attribute Protocol (ATT). This is also referred to as GATT/ATT. ATT is optimized to run on BLE devices. To this end, it uses as few bytes as possible. Each attribute is uniquely identified by a Universally Unique Identifier (UUID), which is a standardized 128-bit format for a string ID used to uniquely identify information. The *attributes* transported by ATT are formatted as *characteristics* and *services*.\n\n> - **Characteristic** \u2014A characteristic contains a single value and 0-n descriptors that describe the characteristic's value. A characteristic can be thought of as a type, analogous to a class. \n\n> - **Descriptor** \u2014Descriptors are defined attributes that describe a characteristic value. For example, a descriptor might specify a human-readable description, an acceptable range for a characteristic's value, or a unit of measure that is specific to a characteristic's value.\n\n> - **Service** \u2014A service is a collection of characteristics. For example, you could have a service called \"Heart Rate Monitor\" that includes characteristics such as \"heart rate measurement.\" You can find a list of existing GATT-based profiles and services on [bluetooth.org](https://www.bluetooth.org/en-us/specification/adopted-specifications).\n\n\u3053\u308c\u3089\u306fBLE\u306e\u30ad\u30fc\u30ef\u30fc\u30c9\u3068\u30b3\u30f3\u30bb\u30d7\u30c8\u3067\u3042\u308b. \n\n - **Generic Attribute Profile (GATT)** GATT\u30d7\u30ed\u30d5\u30a1\u30a4\u30eb\u306fBLE\u901a\u4fe1\u306b\u3088\u308b\u30c7\u30fc\u30bf\u306e\u9001\u4fe1\u3084\u65e2\u77e5\u306e\"attributes\"\u306e\u65ad\u7247\u3092\u53d7\u4fe1\u3059\u308b\u305f\u3081\u306e\u57fa\u672c\u4ed5\u69d8\u3067\u3042\u308b. \u5168\u3066\u306eBLE\u30a2\u30d7\u30ea\u30b1\u30fc\u30b7\u30e7\u30f3\u306e\u30d7\u30ed\u30d5\u30a1\u30a4\u30eb\u306fGATT\u30d7\u30ed\u30d5\u30a1\u30a4\u30eb\u304c\u30d9\u30fc\u30b9\u306b\u306a\u308b.\n\t - Bluetooth SIG\u3067\u306f\u3044\u3064\u304f\u304b\u306eBLE[\u30d7\u30ed\u30d5\u30a1\u30a4\u30eb](https://www.bluetooth.org/en-us/specification/adopted-specifications)\u304c\u5b9a\u7fa9\u3055\u308c\u3066\u3044\u308b. \u30d7\u30ed\u30d5\u30a1\u30a4\u30eb\u306f\u30c7\u30d0\u30a4\u30b9\u304c\u7279\u5b9a\u306e\u30a2\u30d7\u30ea\u30b1\u30fc\u30b7\u30e7\u30f3\u3067\u3069\u306e\u3088\u3046\u306b\u52d5\u4f5c\u3059\u308b\u304b\u306e\u4ed5\u69d8\u3067\u3042\u308b. \u30c7\u30d0\u30a4\u30b9\u306f\u8907\u6570\u306e\u30d7\u30ed\u30d5\u30a1\u30a4\u30eb\u3092\u5b9f\u88c5\u3067\u304d\u308b\u70b9\u306b\u306f\u6ce8\u610f\u3059\u308b\u3053\u3068. \u4f8b\u3048\u3070\u3042\u308b\u30c7\u30d0\u30a4\u30b9\u306f\u5fc3\u62cd\u6570\u30e2\u30cb\u30bf\u30fc\u3068\u96fb\u529b\u6d88\u8cbb\u30e2\u30cb\u30bf\u30fc\u3092\u542b\u3080\u3053\u3068\u304c\u3067\u304d\u308b. \n\n - **Attribute Protocal (ATT)** - GATT\u306fAttribute Prorotocol(ATT)\u306e\u4e0a\u4f4d\u306b\u7d44\u307f\u8fbc\u307e\u308c\u308b. \u3053\u308c\u306fGATT/ATT\u3068\u547c\u3070\u308c\u308b. This applies to the BLE connection itself. The device in the central role scans, looking for advertisement, and the device in the peripheral role makes the advertisement.\n> - GATT server vs. GATT client. This determines how two devices talk to each other once they've established the connection.\n\n> To understand the distinction, imagine that you have an Android phone and an activity tracker that is a BLE device. The phone supports the central role; the activity tracker supports the peripheral role (to establish a BLE connection you need one of each\u2014two things that only support peripheral couldn't talk to each other, nor could two things that only support central).\n\n> Once the phone and the activity tracker have established a connection, they start transferring GATT metadata to one another. Depending on the kind of data they transfer, one or the other might act as the server. For example, if the activity tracker wants to report sensor data to the phone, it might make sense for the activity tracker to act as the server. If the activity tracker wants to receive updates from the phone, then it might make sense for the phone to act as the server.\n\n> In the example used in this document, the Android app (running on an Android device) is the GATT client. The app gets data from the GATT server, which is a BLE heart rate monitor that supports the [Heart Rate Profile](http://developer.bluetooth.org/TechnologyOverview/Pages/HRP.aspx).
