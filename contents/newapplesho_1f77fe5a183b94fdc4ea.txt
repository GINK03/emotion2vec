Module System for a General-Purpose Blocks Language\nYoshiki Ohshima, Jens M\u00f6nig, John Maloney\n\nOur team is developing GP, a new blocks language that aims to be\nbeginner-friendly, like Scratch, yet capable of scaling up to support\nlarger applications. We hope to allow a worldwide community of users\nto share projects, sprites, and code libraries and to create new ones\nusing a mash-up development style. To support such easy sharing and\nreuse, GP incorporates a strong notion of modularity. Modularity\nallows components created by different users at different times to\ninteroperate without worrying about conflicts. However, we do not want\nmodularity to add to the burden of the beginning programmer; we\u2019d like\nGP\u2019s module features to stay out of the way until they are needed. In\nthis paper, we present the key ideas around the GP module system. A\nmodule in GP is a unit of encapsulated code and data. It exports a\nselected set of classes, functions, and variables, provides a\nnamespace for the code inside, and may include private helper classes,\nfunctions, and variables. A module can also extend system classes with\nadditional methods. Such extensions can be used freely within the\nmodule but are not visible outside it. Modules can be saved and\nre-loaded, allowing modules to store user-created projects, sprites,\nand code
